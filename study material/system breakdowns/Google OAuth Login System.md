## 1. User Journey (Narrative)

### Preconditions

- User has a Google account.
- Relying Party (RP) — the third-party app — has registered with Google Cloud Console, obtained a `client_id` and `client_secret`, and configured allowed redirect URIs.
- RP's backend is running. Frontend is served via HTTPS.

### Step-by-Step Narrative

#### T+0ms — User clicks "Sign in with Google"

The user sees a button. Under the hood, the frontend JavaScript renders an anchor tag or fires a `window.location.assign()` call. Nothing has gone over the wire yet. The browser hasn't spoken to Google at all.

The RP frontend constructs an authorization URL. This construction happens entirely client-side:

```
https://accounts.google.com/o/oauth2/v2/auth
  ?response_type=code
  &client_id=<RP_CLIENT_ID>
  &redirect_uri=https://app.example.com/auth/callback
  &scope=openid%20email%20profile
  &state=<CSRF_TOKEN>
  &code_challenge=<PKCE_CHALLENGE>
  &code_challenge_method=S256
  &access_type=offline
  &prompt=consent
```

The `state` parameter is a cryptographically random value (128-bit minimum) generated by the RP frontend or backend, stored in a `HttpOnly` session cookie or server-side session before the redirect fires. It will be validated after Google redirects back.

If PKCE is used (mandatory for public clients, strongly recommended for confidential clients), the RP generates:

```
code_verifier = base64url(random(32 bytes))  // 256-bit random
code_challenge = base64url(SHA256(code_verifier))
```

The `code_verifier` is stored in the user's session (server side) or `sessionStorage` (client side, less ideal). Only the `code_challenge` goes to Google.

#### T+0ms to T+~50ms — Browser navigates to Google

The browser issues a GET to `accounts.google.com`. This triggers full DNS + TCP + TLS (detailed in Section 2). The browser follows the redirect.

#### T+~50–300ms — Google renders its login UI

Google's authorization server receives the request. It validates:

1. `client_id` exists in its registry.
2. `redirect_uri` exactly matches one of the pre-registered URIs for that client.
3. `scope` values are valid.
4. `response_type=code` is permitted for this client.

If the user is already authenticated with Google (active Google session cookie exists), Google may skip the username/password UI and jump straight to consent. If not, it renders the login form — served from Google's own origin, so the RP never sees credentials.

The user enters credentials. Google validates them against its internal identity store (this involves bcrypt/scrypt password hashing, 2FA if enabled, risk scoring, device trust evaluation, etc. — all opaque to the RP).

#### T+~1–3s — Consent screen (if first-time or forced)

Google presents the OAuth consent screen showing:
- The RP's registered name and logo
- Requested scopes (`openid`, `email`, `profile`)
- Legal links

User clicks "Allow."

#### T+~3s — Google issues authorization code and redirects back

Google generates a short-lived (typically **60 seconds** TTL), single-use authorization code. This is an opaque, cryptographically random token bound to:
- The `client_id`
- The `redirect_uri`
- The authenticated user's identity
- The approved scopes
- The `code_challenge` (PKCE)

Google issues an HTTP 302 redirect to the RP's `redirect_uri`:

```
HTTP/1.1 302 Found
Location: https://app.example.com/auth/callback
  ?code=4/0AfJohXn...
  &state=<CSRF_TOKEN_ECHOED_BACK>
```

The browser follows this redirect. The authorization code is visible in the URL query string — this is intentional and safe because the code is single-use and short-lived. However, it may appear in browser history and server access logs on the RP side, which is why codes must be exchanged immediately.

#### T+~3–3.1s — RP backend receives the callback

The RP's callback endpoint (`/auth/callback`) receives the GET request. It performs server-side steps:

1. **Validate `state`**: Compare the echoed `state` parameter to the value stored in the user's session. If mismatch → abort, possible CSRF attempt.
2. **Extract `code`**: Pull the authorization code from the query string.
3. **Perform token exchange**: Make a server-to-server POST to Google's token endpoint.

#### T+~3.1–3.3s — Token exchange (backend-to-Google)

The RP's backend makes a POST request from its server to:

```
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=4/0AfJohXn...
&redirect_uri=https://app.example.com/auth/callback
&client_id=<CLIENT_ID>
&client_secret=<CLIENT_SECRET>
&code_verifier=<PKCE_VERIFIER>
```

This exchange happens server-to-server. The user's browser is not involved. The `client_secret` never leaves the RP's backend. Google validates:

1. Code exists, has not been used, has not expired.
2. `client_id` matches the code's bound client.
3. `redirect_uri` matches exactly.
4. `client_secret` authenticates the client.
5. `code_verifier` → SHA256 → matches stored `code_challenge` (PKCE).

#### T+~3.3s — Google returns tokens

Google responds with a JSON body:

```json
{
  "access_token": "ya29.a0AfH...",
  "expires_in": 3600,
  "refresh_token": "1//0gLm...",
  "scope": "openid email profile",
  "token_type": "Bearer",
  "id_token": "eyJhbGci..."
}
```

Three tokens returned. Three distinct purposes:

- **`id_token`** (JWT): Proves *who the user is* (authentication). Contains claims: `sub`, `email`, `name`, `aud`, `iss`, `iat`, `exp`, `nonce`.
- **`access_token`**: Proves *what the RP can do on the user's behalf* (authorization). Opaque to the RP — only Google APIs understand it.
- **`refresh_token`**: Long-lived token to mint new access tokens without user interaction.

#### T+~3.3–3.4s — RP validates the ID token and creates a session

The RP backend:

1. Fetches Google's JWKS from `https://www.googleapis.com/oauth2/v3/certs` (or uses cached keys) to get the public key.
2. Verifies the JWT signature using RS256.
3. Validates: `iss` == `https://accounts.google.com`, `aud` == RP's `client_id`, `exp` > now, `nonce` matches (replay protection).
4. Extracts `sub` (Google's stable, permanent user ID) and email.
5. Looks up or creates a user record in RP's database using `sub` as the canonical identifier (not email — email can change).
6. Creates a server-side session (or issues its own JWT) for the user.
7. Stores `access_token` and `refresh_token` encrypted in the database if RP needs to call Google APIs on behalf of the user.

#### T+~3.4s — RP redirects user to authenticated state

RP sets a session cookie (`HttpOnly; Secure; SameSite=Lax`) and redirects to the app's home page or the originally requested resource. From the user's perspective: they clicked a button, saw Google's familiar UI, and are now logged in. Total elapsed: ~3–5 seconds.

## 2. Network Layer Flow

### DNS Resolution

When the browser navigates to `https://accounts.google.com/o/oauth2/v2/auth`:

```
Browser Cache → OS Cache → /etc/hosts → Recursive Resolver (ISP or 8.8.8.8)
     → Root Nameserver → .com TLD NS → Google's Authoritative NS
     → Returns A/AAAA record
```

In detail:

1. **Browser DNS cache**: Chrome caches DNS with a 60s TTL minimum. Cache hit → skip to TCP.
2. **OS stub resolver**: Queries the configured DNS server (likely ISP's recursive resolver or a corporate DNS).
3. **Recursive resolver**: Has its own cache. On miss, performs iterative resolution:
   - Queries a root nameserver (one of 13 logical roots, anycast, hundreds of physical nodes).
   - Gets referral to `.com` TLD nameservers (operated by Verisign).
   - Gets referral to Google's authoritative nameservers (`ns1.google.com` through `ns4.google.com`).
   - Queries Google's NS, gets IP (e.g., `142.250.80.78`).
4. **DNSSEC**: Google's zones are DNSSEC-signed. Recursive resolvers that support DNSSEC will validate the chain of trust. If validation fails, the query is dropped.
5. **TTL**: Google typically sets short TTLs (300s) on auth endpoints to allow rapid failover across their global load balancer fleet.

**Failure mode**: DNS poisoning, DNS-over-UDP hijacking (mitigated by DNSSEC and DoH/DoT), resolver unavailability.

**Latency contribution**: 0ms (cache hit) to ~50ms (cold resolution from scratch).

### TCP 3-Way Handshake

Once the browser has the IP address:

```
Client                              Google LB (anycast IP)
  |                                       |
  |  SYN (seq=x)                          |
  |-------------------------------------->|
  |                                       |
  |  SYN-ACK (seq=y, ack=x+1)            |
  |<--------------------------------------|
  |                                       |
  |  ACK (ack=y+1)                        |
  |-------------------------------------->|
  |  [TCP connection established]         |
```

Key details:
- Google operates an anycast load balancer (Maglev/Andromeda) — the SYN packet is routed to the nearest Google PoP based on BGP routing. The user may be talking to a Google edge node, not a datacenter.
- **TCP Fast Open (TFO)**: On repeat connections, TFO allows data to be sent with the SYN packet, eliminating one round-trip.
- **Initial Congestion Window**: Google uses an ICW of 10 MSS (as per RFC 6928), meaning the first 14KB of data can flow without waiting for ACKs.
- **RTT**: Client-to-Google edge is typically 5–30ms within a region. This RTT directly contributes to TLS overhead.

### TLS 1.3 Handshake

TLS 1.3 (mandated for Google's auth endpoints) drastically reduces handshake latency:

```
Client                                         Server
  |                                               |
  |  ClientHello                                  |
  |  [TLS version, cipher suites,                 |
  |   key_share (ephemeral ECDH pub key),          |
  |   SNI: accounts.google.com,                   |
  |   ALPN: h2,                                   |
  |   session_ticket (for 0-RTT)]                 |
  |---------------------------------------------->|
  |                                               |
  |  ServerHello                                  |
  |  [selected cipher suite,                      |
  |   server's ephemeral ECDH pub key]            |
  |  {EncryptedExtensions}                        |
  |  {Certificate}                                |
  |  {CertificateVerify}                          |
  |  {Finished}                                   |
  |<----------------------------------------------|
  |                                               |
  |  {Finished}                                   |
  |  [Application Data — HTTP request]            |
  |---------------------------------------------->|
```

**What's happening cryptographically:**

1. **Key Exchange**: Both sides contribute an ephemeral ECDH key share (typically X25519). The shared secret is computed: `SharedSecret = ClientPrivKey * ServerPubKey` (elliptic curve Diffie-Hellman). This shared secret is never transmitted.
2. **Key Derivation**: The shared secret is passed through HKDF (HMAC-based Key Derivation Function) to produce separate keys for each direction and each purpose (handshake encryption, application data encryption).
3. **Cipher Suite**: Google prefers `TLS_AES_128_GCM_SHA256` or `TLS_CHACHA20_POLY1305_SHA256`. Both are AEAD ciphers — they provide confidentiality and integrity in one pass.
4. **Certificate**: Google presents an X.509 cert for `accounts.google.com` signed by Google Trust Services (GTS). The browser validates: chain of trust → trusted root CA, hostname match, `notAfter` date, OCSP staple or CRL for revocation.
5. **Forward Secrecy**: Because ECDH keys are ephemeral (generated fresh per session), compromising the server's long-term private key does NOT expose past sessions. This is Perfect Forward Secrecy (PFS).
6. **0-RTT**: On repeat connections, clients can send application data immediately in the first flight using a session ticket (a resumption secret encrypted by the server). **Security caveat**: 0-RTT data is not protected against replay attacks; Google limits what operations are safe to perform on 0-RTT data (typically safe for idempotent GETs).

**ALPN**: The client advertises `h2` (HTTP/2). Google's auth servers use HTTP/2 or HTTP/3 (QUIC). HTTP/2 multiplexes multiple streams over a single TCP connection, eliminating head-of-line blocking at the HTTP level.

**CT (Certificate Transparency)**: Google's own certs are logged in public CT logs. Browsers validate CT by checking for SCTs (Signed Certificate Timestamps) embedded in the cert or stapled via OCSP/TLS extension.

## 3. Application Layer Flow

### Initial Authorization Request (Browser → Google)

**Method**: GET  
**Protocol**: HTTPS/HTTP2  
**Endpoint**: `https://accounts.google.com/o/oauth2/v2/auth`

**Headers sent by browser:**
```http
GET /o/oauth2/v2/auth?response_type=code&client_id=...&redirect_uri=...&scope=openid+email+profile&state=...&code_challenge=...&code_challenge_method=S256 HTTP/2
Host: accounts.google.com
User-Agent: Mozilla/5.0 (...)
Accept: text/html,application/xhtml+xml,...
Accept-Language: en-US,en;q=0.9
Accept-Encoding: gzip, deflate, br
Cookie: CONSENT=...; SID=...; SSID=...; HSID=...; APISID=...
Sec-Fetch-Dest: document
Sec-Fetch-Mode: navigate
Sec-Fetch-Site: cross-site
```

Google's cookies (`SID`, `SSID`, `HSID`) are Google-wide session cookies set on `.google.com`. They tell Google's auth server if the user is already signed in to a Google account. These cookies use the `__Secure-` prefix and are `HttpOnly; Secure; SameSite=None` (because they need to be sent cross-site during the OAuth flow).

**How Google parses the request:**
1. URL query string is parsed. Each parameter is URL-decoded.
2. `client_id` is looked up in Google's OAuth registry (sharded database/key-value store).
3. `redirect_uri` is compared against the registered list using exact string matching (not prefix matching — a critical security requirement).
4. `scope` values are split on space/`%20` and validated individually against allowed scopes.
5. `state` is an opaque passthrough — Google doesn't validate its content, only echoes it back.
6. `code_challenge_method` must be `S256` (plain is deprecated and rejected for most clients).

### Callback Request (Browser → RP)

After Google redirects, the browser makes:

```http
GET /auth/callback?code=4%2F0AfJohXn...&state=abc123xyz HTTP/2
Host: app.example.com
Cookie: session_id=...; csrf_token=...
Referer: https://accounts.google.com/
```

**Important**: The `Referer` header will contain `accounts.google.com`. The RP must be careful not to log the full referrer URL, as query params from the Google auth page can appear here.

The RP's web framework parses `code` and `state` from the query string. The `code` value is URL-decoded (it contains `/` encoded as `%2F`). Standard parameter parsing handles this — but a naive string split on `?` and `&` can break if the code contains `&` in its encoded form.

### Token Exchange Request (RP Backend → Google)

```http
POST /token HTTP/2
Host: oauth2.googleapis.com
Content-Type: application/x-www-form-urlencoded
Content-Length: 287
User-Agent: google-api-python-client/2.0

grant_type=authorization_code&
code=4%2F0AfJohXn...&
redirect_uri=https%3A%2F%2Fapp.example.com%2Fauth%2Fcallback&
client_id=123456789-abc.apps.googleusercontent.com&
client_secret=GOCSPX-...&
code_verifier=dBjftJeZ4CVP...
```

This is a `application/x-www-form-urlencoded` POST, not JSON. The `client_secret` is transmitted here — which is why this request MUST happen server-to-server over TLS, never from the browser.

**Google's token endpoint response:**

```http
HTTP/2 200
Content-Type: application/json; charset=utf-8
Cache-Control: no-store
Pragma: no-cache
X-Content-Type-Options: nosniff
X-Frame-Options: SAMEORIGIN
X-XSS-Protection: 1; mode=block

{
  "access_token": "ya29.a0AfH6SMB...",
  "expires_in": 3600,
  "refresh_token": "1//0gLm...",
  "scope": "openid email https://www.googleapis.com/auth/userinfo.email https://www.googleapis.com/auth/userinfo.profile",
  "token_type": "Bearer",
  "id_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6IjZmOTc3N2..."
}
```

**Response headers of note:**
- `Cache-Control: no-store` — prevent tokens from being cached by intermediaries.
- `no-cache` via `Pragma` header — legacy HTTP/1.0 compat.

### ID Token (JWT) Structure and Parsing

The `id_token` is a base64url-encoded JWT with three dot-separated parts:

```
Header.Payload.Signature
```

**Header** (decoded):
```json
{
  "alg": "RS256",
  "kid": "6f9777d...",
  "typ": "JWT"
}
```

**Payload** (decoded):
```json
{
  "iss": "https://accounts.google.com",
  "azp": "123456789-abc.apps.googleusercontent.com",
  "aud": "123456789-abc.apps.googleusercontent.com",
  "sub": "110169484474386276334",
  "email": "user@gmail.com",
  "email_verified": true,
  "name": "Jane Doe",
  "picture": "https://lh3.googleusercontent.com/a/...",
  "given_name": "Jane",
  "family_name": "Doe",
  "locale": "en",
  "iat": 1700000000,
  "exp": 1700003600,
  "nonce": "abc123xyz"
}
```

**Signature**: RS256 — the JWT payload is signed with Google's private RSA key (2048-bit). The RP verifies using the corresponding public key fetched from `https://www.googleapis.com/oauth2/v3/certs`.

**Critical parsing rules for the RP:**
1. Never trust the `alg` field in the JWT header without also constraining which algorithms your library accepts. Attackers can try to substitute `alg=none` (MITM) or `alg=HS256` with the public key as the HMAC secret.
2. `sub` is the stable user identifier. `email` can change.
3. `exp` must be checked against current server time. Clock skew allowance should be at most 60 seconds.
4. `aud` must exactly match the RP's `client_id`.
5. `iss` must be exactly `https://accounts.google.com` or `accounts.google.com` (both are valid per Google's spec).

## 4. Backend Architecture

### Services Involved (RP Side)

```
                    ┌─────────────────────────────────────────────┐
                    │              RP Application Stack            │
                    │                                             │

                    │       │                                     │
                    │       ├──> Web Server / API Gateway         │
                    │       │      │                              │
                    │       │      ├──> Auth Service              │
                    │       │      │      │                       │
                    │       │      │      ├──> Session Store      │
                    │       │      │      │    (Redis)            │
                    │       │      │      │                       │
                    │       │      │      ├──> User DB            │
                    │       │      │      │    (PostgreSQL)       │
                    │       │      │      │                       │
                    │       │      │      └──> Token Vault        │
                    │       │      │           (Encrypted store   │
                    │       │      │            for refresh tokens)│
                    │       │      │                              │
                    │       │      └──> Business Logic Services   │
                    │                                             │
                    └─────────────────────────────────────────────┘
                                        │
                              Google OAuth APIs
                          (accounts.google.com,
                           oauth2.googleapis.com,
                           www.googleapis.com)
```

### Auth Service Responsibilities

The Auth Service is the most security-critical component. It handles:

1. **State generation and validation**: Generates `state` (CSRF token) before redirect; validates on callback.
2. **PKCE**: Generates `code_verifier`, derives `code_challenge`, stores verifier in session, sends challenge to Google.
3. **Token exchange**: Issues the server-to-server POST to Google's token endpoint.
4. **JWT validation**: Verifies `id_token` signature, claims, expiry.
5. **User provisioning/lookup**: Queries the User DB by `sub`. Creates user record if new (JIT provisioning).
6. **Session creation**: Writes a session record to Redis, sets a session cookie.
7. **Token storage**: Encrypts and stores the refresh token in the DB if offline access is needed.

### Session Store (Redis)

Sessions are stored in Redis as key-value pairs:

```
Key:   session:<session_id>
Value: {
  "user_id": "db_user_id_123",
  "google_sub": "110169484474386276334",
  "email": "user@gmail.com",
  "created_at": 1700000000,
  "last_seen": 1700000000,
  "ip": "203.0.113.42",
  "user_agent_hash": "sha256:abc...",
  "access_token": "<encrypted>",
  "access_token_exp": 1700003600
}
TTL: 86400 (24 hours, sliding window)
```

Session IDs are 256-bit random values (32 bytes of CSPRNG output), base64url-encoded. Redis is configured with `requirepass` and TLS. The auth service connects to Redis over a private subnet — never exposed to the internet.

**Why Redis for sessions?**
- O(1) reads/writes for session lookup on every request.
- TTL management is native.
- Supports atomic operations (compare-and-swap for session invalidation).
- Horizontal scaling via Redis Cluster or Redis Sentinel for HA.

### Database Interactions

**User table** (PostgreSQL):
```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  google_sub VARCHAR(255) UNIQUE NOT NULL,
  email VARCHAR(255) NOT NULL,
  email_verified BOOLEAN NOT NULL DEFAULT false,
  display_name VARCHAR(255),
  picture_url TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_login_at TIMESTAMPTZ,
  is_active BOOLEAN NOT NULL DEFAULT true
);

CREATE UNIQUE INDEX idx_users_google_sub ON users(google_sub);
CREATE INDEX idx_users_email ON users(email);
```

The Auth Service queries by `google_sub` (indexed, unique) — a single indexed lookup. If no row found, it inserts a new user (JIT provisioning). This is done in a transaction with `ON CONFLICT (google_sub) DO UPDATE` to handle race conditions where two requests arrive simultaneously for the same new user.

**Refresh token table**:
```sql
CREATE TABLE oauth_refresh_tokens (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id),
  token_encrypted BYTEA NOT NULL,       -- AES-256-GCM encrypted
  token_hash VARCHAR(64) NOT NULL,      -- SHA-256 of plaintext for lookup
  scope TEXT NOT NULL,
  issued_at TIMESTAMPTZ NOT NULL,
  last_used_at TIMESTAMPTZ,
  revoked_at TIMESTAMPTZ,
  UNIQUE(token_hash)
);
```

Refresh tokens are never stored in plaintext. They are encrypted at rest with a key stored in a secrets manager (AWS KMS, HashiCorp Vault, GCP KMS). The plaintext token is needed only when calling Google to refresh the access token.

### Async Flows

The core OAuth callback is synchronous — the user is waiting. However, several side effects are triggered asynchronously:

- **Audit log**: `user_logged_in` event published to a message queue (Kafka/SQS). Consumed by an audit log service that writes to an append-only store (S3 + Athena, or a SIEM like Splunk).
- **Account provisioning**: For enterprise SSO, additional group/role assignment may be triggered as an async job.
- **Notification**: First-login email sent via async worker (email queue).
- **Analytics**: Login events tracked asynchronously in a data warehouse.

Why async? The user should not wait for email delivery or analytics writes. These are not on the critical path of authentication.

### Caching Layers

**Google's JWKS (JSON Web Key Set):**
- URL: `https://www.googleapis.com/oauth2/v3/certs`
- The RP caches this response. Google sets a `Cache-Control: max-age=...` header (typically 6–24 hours).
- The cache is keyed by the `kid` (Key ID) in the JWT header. When the RP sees a `kid` it doesn't have cached, it fetches the JWKS immediately.
- Google rotates keys roughly every few weeks. The RP must handle key rotation gracefully — cache the full JWKS, not just one key.

**CDN (CloudFront/Fastly):**
- The RP's static assets (JS, CSS, images) are served from a CDN.
- The `/auth/callback` endpoint is NEVER behind a CDN cache — it handles stateful session creation. CDN passthrough must be configured for `/auth/*`.

## 5. Authentication & Authorization Flow

### The Three-Phase Model

OAuth 2.0 separates three distinct security operations that are often conflated:

1. **Authentication** (who is the user?): Performed by Google. Proven to the RP via the signed `id_token`.
2. **Authorization** (what can the user do at Google on behalf of the RP?): The access token grants the RP permission to call specific Google APIs.
3. **Session** (is this browser associated with an authenticated user?): The RP's own session mechanism, independent of Google's tokens.

### Trust Boundaries

```
┌─────────────────────────────────────────────────────────┐
│                     TRUST BOUNDARY 1                    │
│  User ↔ Google (Google fully controls this boundary)    │
│  - User authenticates to Google with password/2FA       │
│  - Google issues authorization code                      │
│  - RP trusts Google's assertions in the id_token        │
└─────────────────────────────────────────────────────────┘
                           │
                     authorization
                        code (via
                       browser URL)
                           │
┌─────────────────────────────────────────────────────────┐
│                     TRUST BOUNDARY 2                    │
│  RP Backend ↔ Google (authenticated with client_secret) │
│  - Code exchanged for tokens over mTLS-like channel     │
│  - Google authenticates the RP using client_secret      │
│  - This channel never involves the browser              │
└─────────────────────────────────────────────────────────┘
                           │
                       id_token,
                     access_token,
                     refresh_token
                           │
┌─────────────────────────────────────────────────────────┐
│                     TRUST BOUNDARY 3                    │
│  Browser ↔ RP (RP's own session mechanism)              │
│  - RP sets HttpOnly session cookie                      │
│  - Browser sends cookie on every request to RP          │
│  - RP validates session on each request                 │
└─────────────────────────────────────────────────────────┘
```

### Token Validation Pipeline (RP Side)

Every incoming request to the RP with a session cookie:

```
Request
  │
  ▼
Extract session_id from cookie
  │
  ▼
Lookup session_id in Redis
  │
  ├── MISS ──> Return 401 / redirect to login
  │
  └── HIT ──>
        │
        ▼
       Check session.created_at + TTL > now
        │
        ├── EXPIRED ──> Delete from Redis, return 401
        │
        └── VALID ──>
              │
              ▼
             Check session.ip (optional, for high-risk operations)
              │
              ▼
             Load user from DB (or use session-cached user data)
              │
              ▼
             Check user.is_active == true
              │
              ├── FALSE ──> Return 403
              │
              └── TRUE ──> Attach user to request context, proceed
```

### Access Token Lifecycle

The `access_token` expires in 3600 seconds (1 hour). If the RP needs to call Google APIs:

1. Before making a Google API call, check `access_token_exp` in session.
2. If expired (or within 60s of expiry), use the `refresh_token` to get a new `access_token`:

```http
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=1//0gLm...
&client_id=...
&client_secret=...
```

3. Google returns a new `access_token` (and sometimes a new `refresh_token` — always store the latest one, as Google may rotate them).
4. Update the session and DB with the new tokens.

**Refresh token revocation:** If the user revokes access in their Google account settings, the refresh token becomes invalid. The next refresh attempt will return `invalid_grant`. The RP must handle this error by forcing re-authentication and cleaning up the stored token.

### Session Invalidation

When a user logs out:
1. RP deletes the session from Redis.
2. RP clears the session cookie (by setting it with `expires=epoch`).
3. Optionally, RP revokes the Google tokens: `POST https://oauth2.googleapis.com/revoke?token=<refresh_token>`.
4. Publish `user_logged_out` event to audit queue.

Critically: logging out of the RP does NOT log the user out of Google. The user's Google session (`SID` cookie on google.com) remains active. This is intentional in OAuth — the RP has no authority over Google's session.

## 7. Security Controls

### Encryption in Transit

- All traffic between browser and both Google and RP uses TLS 1.2 minimum; TLS 1.3 preferred.
- RP-to-Google server-to-server calls use TLS 1.3 with certificate pinning in some implementations (or at minimum, validation of Google's cert chain).
- Redis connections encrypted in transit (TLS). Connections are authenticated with Redis `AUTH` command using a long secret password.
- PostgreSQL connections use SSL mode `verify-full` (client validates server cert, server may validate client cert).
- Internal service-to-service communication within RP's VPC: TLS with mutual TLS (mTLS) using short-lived certificates (SPIFFE/SPIRE or AWS ACM PCA).

### Encryption at Rest

- Refresh tokens: AES-256-GCM encrypted before storage. Encryption key stored in AWS KMS / HashiCorp Vault. Key rotation on a schedule (annually, or on suspected compromise).
- User PII in DB: Encrypted at rest via database-level encryption (PostgreSQL TDE or cloud-managed: AWS RDS encryption using KMS).
- Redis: Encrypted at rest (AWS ElastiCache with encryption-at-rest enabled, uses AES-256).
- Audit logs: Encrypted at rest in S3 (SSE-KMS).
- `client_secret`: Never stored in code, config files, or environment variables in plaintext. Stored in secrets manager (AWS Secrets Manager, HashiCorp Vault). Retrieved at application startup or on demand, cached in memory only, never logged.

### Input Validation

- **`state` parameter**: The RP generates it (CSPRNG, 128-bit min). On callback, exact comparison using `hmac.compare_digest()` (constant-time) to prevent timing attacks.
- **`code` parameter**: Opaque; passed directly to Google's token endpoint without interpretation. Length sanity check (Google codes are typically <500 chars).
- **`redirect_uri`**: Registered in advance. RP must send exactly the registered URI. Google enforces exact match.
- **JWT claims**: Every claim validated. Library used must be well-audited (e.g., `python-jose`, `jsonwebtoken`). `alg` constrained to `RS256` only — the library must not honor JWT header `alg` blindly.
- **SQL**: Parameterized queries only. No string interpolation into SQL.
- **Session data**: User-controlled data from the JWT (email, name) treated as untrusted when displayed in the UI — HTML-escaped to prevent stored XSS.

### Access Control

- The `/auth/callback` endpoint only accepts GET requests. Other methods return 405.
- Token exchange endpoint on the RP is not exposed to the browser — it is an internal server-side operation.
- Session cookies have `SameSite=Lax` which prevents them from being sent on cross-site top-level navigations with non-safe methods. `SameSite=Strict` is more restrictive but can break OAuth flows (the callback redirect from Google is a cross-site navigation).
- API endpoints require valid session. Session validation happens in a middleware layer applied to all protected routes.

### Secrets Handling

```
                 ┌─────────────────────────────────────────┐
                 │              Secrets Manager            │
                 │  (AWS Secrets Manager / Vault)          │
                 │                                         │
                 │  oauth_client_secret: GOCSPX-...        │
                 │  redis_password: <256-bit random>        │
                 │  db_password: <256-bit random>           │
                 │  session_signing_key: <256-bit random>   │
                 │  token_encryption_key: <256-bit AES key> │
                 └────────────────┬────────────────────────┘
                                  │ IAM Role / AppRole
                                  │ authentication
                                  ▼
                          Auth Service process
                          (fetches secrets at
                           startup or on-demand,
                           stores in memory only,
                           never logs)
```

Secrets rotation: Automated via Secrets Manager rotation lambdas. The application must handle secret refresh without restart (e.g., catch auth errors from Google, re-fetch secret, retry once).

## 9. Attack Scenarios (Very Detailed)

### 9.1 — CSRF on OAuth Callback (State Parameter Bypass)

**Attacker Assumptions:**
- Attacker can get the victim to visit a page the attacker controls.
- The RP has a vulnerable implementation that doesn't validate the `state` parameter.

**Step-by-Step Execution:**

1. Attacker initiates an OAuth flow themselves, capturing the authorization URL up to the point Google would redirect back.
2. Attacker obtains the authorization code that Google issues for *their* account (but this isn't the attack vector — the attack is about linking, not code theft).
3. More precisely: The attacker crafts a URL to the RP's callback endpoint containing a valid `code` (obtained by the attacker going through the flow for their own account) plus any `state` value.
4. Victim, already logged into the RP (or about to be), visits attacker-controlled page. Page auto-navigates the victim's browser to `https://app.example.com/auth/callback?code=<ATTACKER_CODE>&state=anything`.
5. If the RP doesn't validate `state`, it exchanges the attacker's code and logs the victim into the **attacker's** Google account on the RP.
6. Attacker then logs in with the same code (race condition) — now both attacker and victim share a session, or the victim is bound to attacker's RP account.

**Where Detection Could Happen:**
- **Correct mitigation**: State validation. The victim's browser has a session with `state=<VICTIM_CSRF_TOKEN>`, which doesn't match the `state` parameter in the attacker-crafted URL → rejected.
- If not caught: Anomaly detection — a user's IP changed drastically; a user who just registered suddenly has a long history (account merge anomaly).

**Why This Works (Without Mitigation):**
The callback URL is unauthenticated. Any valid `code` accepted without binding it to the user's own session flow.

---

### 9.2 — Authorization Code Interception

**Attacker Assumptions:**
- Attacker can observe the victim's HTTP request logs, access the browser history, or is a network MITM.

**Step-by-Step Execution:**

1. Google redirects the victim's browser to `https://app.example.com/auth/callback?code=4/0AfJ...&state=...`.
2. The authorization code is visible in the URL. It appears in:
   - Browser address bar.
   - Browser history.
   - Server access logs on the RP (the RP's nginx/web server logs the full URL including query params).
   - `Referer` header if the victim clicks a link from the callback page to an external site.
3. Attacker with access to server logs extracts the code.
4. Attacker attempts to exchange the code: `POST /token` with the same `code`, their own `client_id`... wait — they don't have the `client_secret`.
5. **However**: If PKCE is not used, and the attacker somehow knows the `client_id` (public) and `client_secret` (compromised), they can exchange the code before the RP does.

**Where Detection Could Happen:**
- Google: If the code is exchanged twice (by RP and attacker), the second exchange returns an error AND Google may invalidate all tokens from that code (code reuse detection triggers session revocation). This is detectable via audit logs.
- RP: `invalid_grant` error on token exchange — indicates code was already used.

**PKCE Mitigation:**
With PKCE, even if the attacker intercepts the code, they cannot exchange it without the `code_verifier` which is only known to the RP's backend (stored in server-side session). The attacker's exchange attempt fails because they can't compute `SHA256(code_verifier) == code_challenge`.

---

### 9.3 — Open Redirect Leading to Token Leakage

**Attacker Assumptions:**
- The RP has an open redirect vulnerability (e.g., `https://app.example.com/redirect?to=https://evil.com`).
- The RP uses `redirect_uri` with a path prefix match instead of exact match (misconfiguration).

**Step-by-Step Execution:**

1. Attacker registers a URL with Google for the RP's `client_id` — impossible if they don't control the RP. But if the RP allows dynamic redirect URIs or uses prefix matching, an attacker can craft: `redirect_uri=https://app.example.com/auth/callback?next=https://attacker.com`.
2. The authorization code is delivered to `https://app.example.com/auth/callback?...&next=https://attacker.com`.
3. RP callback handler exchanges the code (legitimate), then follows `next` parameter for post-login redirect.
4. If `next` is not validated, the victim is redirected to `https://attacker.com`.
5. If the access token is included in the fragment or redirect URL, it's leaked.

**Why This Works:**
Prefix matching on `redirect_uri` is a known mis-implementation. RFC 6749 requires exact matching. The `next` parameter pattern is a classic open redirect, a CWE-601.

**Detection:**
`next` parameter pointing to an external domain logged and flagged. Anomaly: user redirected off-domain post-login.

---

### 9.4 — Refresh Token Theft and Long-Term Account Compromise

**Attacker Assumptions:**
- Attacker has compromised the RP's database (e.g., via SQL injection in another endpoint, or stolen DB credentials).

**Step-by-Step Execution:**

1. Attacker dumps the `oauth_refresh_tokens` table.
2. Tokens are stored encrypted (AES-256-GCM). Attacker needs the encryption key.
3. Encryption key is in AWS KMS. DB credentials don't grant KMS access (IAM separation). **Dead end** — if encryption is implemented correctly.
4. If encryption is NOT implemented (common mistake), refresh tokens are stored in plaintext. Attacker extracts them.
5. Attacker calls `POST https://oauth2.googleapis.com/token` with `grant_type=refresh_token&refresh_token=<STOLEN>&client_id=...&client_secret=...`. Needs the `client_secret`.
6. If `client_secret` is also in the DB or config files, attacker now has persistent access to victims' Google accounts via the Google APIs.

**Why This Is Catastrophic:**
Refresh tokens are long-lived (Google's default: until revoked or ~6 months of inactivity). Attacker has access even after the user changes their RP password. Only Google account password change + revoking RP access in Google security settings stops this.

**Detection:**
Access token usage from unusual IPs/geographies. Google's anomalous access detection. DB dump detected via query rate anomaly or DLP.

---

### 9.5 — JWT Algorithm Confusion Attack (`alg: none` or RS256→HS256)

**Attacker Assumptions:**
- Attacker can forge a JWT if the RP's JWT library doesn't constrain the algorithm.
- Attacker knows the RP's `client_id` and the public key used for RS256 (public, available from JWKS endpoint).

**Step-by-Step Execution:**

**Variant A: `alg: none`**
1. Attacker crafts a JWT with `{"alg":"none","typ":"JWT"}` in the header.
2. Payload: `{"iss":"https://accounts.google.com","aud":"<RP_CLIENT_ID>","sub":"victim_sub","email":"victim@gmail.com","exp":<future>}`.
3. Signature segment is empty: `header.payload.`
4. If the RP's JWT library accepts `alg: none`, the signature check is skipped and the attacker impersonates any user.

**Variant B: RS256 → HS256 confusion**
1. Google uses RS256 (asymmetric). The public key is known (from JWKS endpoint).
2. Attacker crafts a JWT with `{"alg":"HS256"}` in the header.
3. Signs the JWT using the HS256 HMAC with Google's *public key bytes* as the secret.
4. If the RP library uses whatever algorithm the header specifies, it verifies HS256 using the same public key bytes → signature validates.
5. Attacker has forged a JWT.

**Where Detection Could Happen:**
- The attack doesn't generate any traffic to Google. Detection must happen at the RP.
- Log: `alg` field in incoming JWT header. Alert if not `RS256`.

**Mitigation**: Always pass the expected algorithm explicitly to the JWT library. Never accept the algorithm from the token header without validation.

---

### 9.6 — Session Fixation

**Attacker Assumptions:**
- Attacker can set a cookie in the victim's browser (subdomain cookie injection, or physical access, or XSS on any subdomain).

**Step-by-Step Execution:**

1. Attacker creates a pre-known session ID (e.g., `attacker_known_session_id`).
2. Attacker injects a cookie into the victim's browser: `session_id=attacker_known_session_id`.
3. Victim completes the OAuth login flow.
4. If the RP doesn't regenerate the session ID after successful authentication (session fixation vulnerability), the RP binds the authenticated user to `attacker_known_session_id`.
5. Attacker's browser, which has the same `session_id` cookie, now has access to the victim's account.

**Why This Works (Without Mitigation):**
The RP uses the existing session (to store `state` before redirect) and never rotates the session ID upon authentication.

**Mitigation**: Always generate a new session ID upon successful authentication, even if an existing session was used to track pre-auth state.

---

### 9.7 — Supply Chain Attack on RP's Frontend

**Attacker Assumptions:**
- Attacker has compromised an npm package used in the RP's frontend build.

**Step-by-Step Execution:**

1. Attacker compromises a popular npm package (e.g., an analytics library, a UI component).
2. Malicious code added: on page load, exfiltrate `document.cookie` (can't get `HttpOnly` cookies) or scrape the DOM for visible token data, or intercept `fetch`/`XHR` calls.
3. If the RP uses `HttpOnly` cookies for sessions (correct), the attacker can't steal the session cookie directly.
4. However, attacker can still: intercept the authorization code from the URL before it's sent to the backend; inject a phishing form; exfiltrate any non-`HttpOnly` tokens (if the RP stores access tokens in `localStorage` — a mistake).
5. If access tokens are in `localStorage`, attacker runs: `localStorage.getItem('access_token')` and exfiltrates it.

**Why This Is Particularly Dangerous:**
Frontend supply chain attacks are difficult to detect. A malicious package version may pass code review.

**Mitigation**: Subresource Integrity (SRI) for CDN-loaded scripts. Content Security Policy (CSP) to restrict exfiltration destinations. Never store tokens in `localStorage`. Lock npm dependency versions with a lock file.

## 10. Failure Points

### Under Load

**Redis session store becomes a bottleneck:**
- Every request to any authenticated endpoint requires a Redis read (the session lookup).
- At high RPS, Redis becomes a hot path. A single Redis node can handle ~100K ops/sec — this is often sufficient, but at extreme scale (millions of concurrent users), Redis Cluster sharding is required.
- **Failure mode**: Redis ENOMEM (out of memory) — new sessions fail to write; existing sessions may be evicted if `maxmemory-policy` is `allkeys-lru`. Silent session invalidation — users get logged out.

**Google's token endpoint rate limits:**
- Token exchanges are rate-limited per `client_id`. If the RP has a thundering herd of logins simultaneously (e.g., after a scheduled email goes out), token exchange requests may be throttled.
- **Failure mode**: `429 Too Many Requests` from Google. RP must implement exponential backoff with jitter.

**JWKS caching stampede:**
- If Google rotates keys and the RP's JWKS cache expires simultaneously for all instances, every instance makes a request to Google's JWKS endpoint.
- **Failure mode**: Google rate-limits the JWKS fetch. ID token validation fails for all instances simultaneously.
- **Mitigation**: Stagger cache expiration with jitter; use a distributed cache (Redis) for JWKS to prevent multi-instance stampede.

**Database connection pool exhaustion:**
- During high login volume, the Auth Service may exhaust its DB connection pool.
- **Failure mode**: New login requests hang waiting for a connection, eventually timing out. Users see login failures.

### Under Attack

**DDoS on `/auth/callback`:**
- Attacker floods the callback endpoint with malformed or invalid codes.
- Each invalid code causes a backend-to-Google token exchange (network call) before returning an error.
- **Impact**: Amplified load on RP's outbound connections to Google and on RP's backend.
- **Mitigation**: Rate-limit the callback endpoint by IP. Validate `state` before making the token exchange call (state validation is pure local logic — cheap).

**Credential stuffing against Google:**
- Not directly the RP's problem, but if Google accounts are compromised, the RP gets legitimate OAuth flows for attacker-controlled accounts.
- **Detection at RP**: Impossible to distinguish from legitimate logins. Trust signals: account age, login velocity, device fingerprint.

### Common Misconfigurations

| Misconfiguration | Impact | Fix |
|---|---|---|
| `redirect_uri` registered with wildcard or prefix | Open redirect, code hijacking | Exact match, specific paths |
| `state` not validated on callback | CSRF, account takeover | CSPRNG state, compare before exchange |
| PKCE not implemented | Code interception | Implement S256 PKCE |
| `alg` not constrained in JWT validation | JWT forgery | Pin to RS256 only |
| Session ID not rotated post-auth | Session fixation | Regenerate session ID after login |
| Refresh tokens stored in plaintext | Long-term compromise | AES-256-GCM encrypt at rest |
| `client_secret` in environment variable in docker image | Secret exfiltration via image pull | Use secrets manager; inject at runtime |
| Logging full redirect URI (includes `code`) | Code theft from logs | Strip query params in access logs |
| `SameSite=None` on session cookie without necessity | Cross-site request with cookie | Use `SameSite=Lax` or `Strict` |
| Access tokens stored in `localStorage` | XSS token theft | Use `HttpOnly` cookies or in-memory only |
| JWKS not cached | Latency, JWKS endpoint DDoS vector | Cache with proper TTL |
| No nonce in authorization request | Replay attacks on id_token | Generate and validate nonce |

## 11. Mitigations

### Defense-in-Depth Strategy

Security controls are layered. Each layer assumes the one before it may fail.

**Layer 1 — Protocol Hardness:**
- Implement OAuth 2.0 with Authorization Code + PKCE (RFC 7636).
- Use OIDC (OpenID Connect) for authentication assertions — don't build your own.
- Validate all JWT claims per RFC 7519 and Google's OIDC spec.
- Use `nonce` to prevent replay attacks on the `id_token`.
- Constrain `alg` to `RS256` in JWT library configuration.

**Layer 2 — Transport Security:**
- TLS 1.3 everywhere. Disable TLS 1.0 and 1.1.
- HSTS with `includeSubDomains; preload` — prevents downgrade attacks.
- Certificate pinning (or CT enforcement) for RP-to-Google communication.
- Disable HTTP entirely on production domains.

**Layer 3 — Application Security:**
- `state` parameter: 128-bit CSPRNG, stored in server-side session, validated before token exchange using constant-time comparison.
- Session ID rotation on authentication.
- `HttpOnly; Secure; SameSite=Lax` on all session cookies.
- CSRF protection on all state-changing endpoints (double-submit cookie or CSRF token).
- Content Security Policy (CSP) with `script-src 'self'` and explicit allowlist. No `unsafe-inline`.

**Layer 4 — Secret Management:**
- `client_secret` in secrets manager. Never in code, config files, or environment variables.
- Automated secret rotation with application support for rolling update (no downtime).
- Refresh tokens: AES-256-GCM encrypted at rest, separate key in KMS.
- Encryption keys in HSM-backed KMS. Keys never leave HSM in plaintext.

**Layer 5 — Infrastructure:**
- Rate limiting on `/auth/callback`: e.g., 10 requests/minute per IP.
- WAF rules: Block known-bad user agents, anomalous parameter lengths.
- Private subnets for Redis and DB. No public internet access.
- IAM: Principle of least privilege. Auth service role has access only to its own secrets, tables, Redis cluster.
- VPC flow logs. CloudTrail. All API calls audited.

**Layer 6 — Monitoring & Response:**
- Alert on: `invalid_grant` spike (may indicate code theft or token compromise), `state` mismatch spike (CSRF attempts), login velocity anomalies, `alg: none` in any JWT.
- Automated response: Rate limit → CAPTCHA → IP block (progressive).
- Incident playbook: Refresh token compromise → revoke all tokens for affected users, force re-auth, notify users.

### Engineering Tradeoffs

| Control | Security Benefit | Tradeoff |
|---|---|---|
| `SameSite=Strict` | Prevents all cross-site cookie sending | Breaks OAuth callback (cross-site navigation from Google). Use `Lax` instead. |
| PKCE mandatory | Prevents code interception | Slight latency (SHA256 computation, verifier storage). Negligible. |
| Short session TTL (1hr) | Limits session hijacking window | Users must re-authenticate frequently. Balance with sliding TTL. |
| Refresh token rotation | Limits theft impact | Risk of token loss (mobile app crash during rotation — use "grace period" approach). |
| IP binding on session | Prevents session hijacking across IPs | Breaks for mobile users on flaky networks. Soft-bind with alerting only. |
| mTLS on internal services | Prevents lateral movement | Certificate management complexity. Use SPIFFE/SPIRE or cloud-native mesh. |

## 13. Scaling Considerations

### Current Bottlenecks

**1. Google's Token Endpoint:**
- The RP cannot parallelize token exchanges for a single login — it's sequential.
- Google's rate limit is per `client_id`. If the RP has millions of users logging in simultaneously (e.g., start of business day, after a marketing push), rate limiting is possible.
- **Solution**: Implement exponential backoff + jitter. Queue login requests during spikes. Consider multiple OAuth `client_id`s for different product lines (each gets its own rate limit quota).

**2. Redis Session Store:**
- Every authenticated request is a Redis read. At 100K req/sec, Redis is under significant load.
- Redis is single-threaded per shard (for commands). Pipelining and multiplexing help, but there's a ceiling.
- **Horizontal scaling**: Redis Cluster (hash slot partitioning). Session keys are distributed by hash of `session_id` across multiple Redis nodes. Each node handles a subset of sessions.
- **Vertical scaling**: Increase Redis instance size (more RAM, faster CPU). AWS ElastiCache r6g instances with up to 635 GB RAM can hold billions of sessions.

**3. JWKS Cache:**
- If JWKS cache is per-instance (in-memory), each RP instance independently fetches JWKS.
- At 100 instances, key rotation causes 100 simultaneous JWKS fetches.
- **Solution**: Centralized JWKS cache in Redis with leader-election for cache refresh (only one instance fetches at a time). Add jitter to cache TTL.

**4. Database (User Lookup on Login):**
- Login path requires a DB read (user lookup by `google_sub`). Indexed, fast.
- For new user registration (JIT provisioning), requires a DB write.
- **Read scaling**: Read replicas for user lookups. The primary write path (new user inserts) goes to primary.
- **Write scaling**: Sharding by `google_sub` hash (if user table is extremely large). But for most applications, PostgreSQL handles billions of rows comfortably on a modern primary.

### Horizontal vs. Vertical Scaling

**Auth Service (stateless):**
- Horizontally scalable. Each instance is stateless (session state in Redis, not local).
- Can scale to hundreds of instances behind the load balancer.
- Deploy with auto-scaling based on CPU and request latency (not just CPU — high-latency token exchanges may not be CPU-bound).

**Redis:**
- Vertical scaling first (more RAM) — simpler, no sharding complexity.
- Horizontal scaling (Redis Cluster) when single node RAM or ops/sec ceiling is hit.
- Redis Sentinel for HA (automatic failover without sharding).

**PostgreSQL:**
- Vertical scaling (larger instance) for write path.
- Read replicas for read path.
- PgBouncer for connection pooling (prevents connection exhaustion under load).
- For extreme scale: CQRS pattern — separate the login event store from the user query model.

### Consistency Tradeoffs

**Session consistency:**
- Redis Cluster uses eventual consistency between shards. For sessions, this means: after writing a new session, reads from a different Redis node in the cluster may not immediately see it.
- **Impact**: Negligible in practice (the first request after login goes to the same node via consistent hashing; subsequent requests may go elsewhere but the replication lag is microseconds).
- **For session invalidation (logout)**: More critical. A delete on the primary must replicate before the next request hits a replica. Use synchronous replication for session deletes (or accept a small window where a deleted session is still valid).

**User record consistency:**
- JIT user provisioning with `ON CONFLICT DO UPDATE` is idempotent — safe under concurrent writes.
- Read replicas have replication lag (typically <1ms on same-region replica). A user who just signed up and immediately hits a read replica may get a stale read. Solution: Route first-login reads to primary, or use read-your-writes consistency semantics.

## 14. Interview Questions

### Q1: Why does OAuth 2.0 use an authorization code instead of returning the access token directly to the browser?

**Why**: Direct token delivery to the browser (implicit flow, now deprecated) exposes the access token in the URL fragment. The URL can appear in browser history, server logs, and the `Referer` header. The authorization code is short-lived (60s), single-use, and useless without the `client_secret` to exchange it. The actual tokens are issued via a server-to-server call where the `client_secret` proves the RP's identity and the tokens never touch the browser.

**What if PKCE is used?** Even without a `client_secret` (public clients), PKCE ensures that only the entity that initiated the flow (possessing the `code_verifier`) can exchange the code. The code is still delivered via redirect, but interception is defeated.

---

### Q2: What happens if Google rotates its signing keys while an RP has cached the old JWKS? How do you handle this correctly?

When Google rotates keys, new `id_token`s are signed with the new private key (new `kid`). The RP's cache still has the old public key. JWT validation for new logins will fail because the cached public key doesn't match the new signature.

**Correct handling:**
1. JWT validation extracts `kid` from the token header.
2. Look up `kid` in the cached JWKS.
3. If `kid` not found in cache → **immediately refresh JWKS** (don't fail; refresh first, then retry validation once).
4. If `kid` found but signature fails → fail (don't refresh on sig failure — that's a forgery attempt vector).
5. Implement negative caching: after a JWKS refresh, don't immediately refresh again for at least 5 minutes (prevent refresh storms).

---

### Q3: A user completes a Google login, but the RP's callback endpoint returns a 500 error before the session is written. What state is the system in, and what are the failure modes?

**State analysis:**
- Google has issued and delivered an authorization code.
- The RP has received the code (it hit the callback endpoint).
- Depending on where the 500 occurred:
  - **Before token exchange**: Code is still valid at Google. User can re-attempt the OAuth flow. The old code will expire after 60s.
  - **After token exchange but before session write**: Google has issued tokens. The authorization code is now consumed (single-use). The access token and refresh token exist at Google but are not stored in the RP's DB. If the user retries the OAuth flow, Google issues new tokens.
- **Risk**: If the RP retries the token exchange with the same (now-consumed) code, it gets `invalid_grant`. The RP must detect this and redirect to re-initiate the OAuth flow.

---

### Q4: Explain how `SameSite=Lax` on the session cookie interacts with the OAuth redirect flow. Why not `SameSite=Strict`?

`SameSite=Strict` means the cookie is **never sent on cross-site requests**, including top-level navigations from external sites. The OAuth callback is a cross-site, top-level navigation (Google → RP). With `SameSite=Strict`, the session cookie that holds the `state` value for CSRF validation would **not be sent** on the callback request. The RP can't validate `state` without the session. Strict breaks the OAuth flow.

`SameSite=Lax` allows cookies on cross-site **top-level navigations using safe HTTP methods** (GET, HEAD). The OAuth callback is a GET request following a redirect — exactly this case. The session cookie is sent, `state` can be validated.

**What if we want Strict?** Some implementations store `state` in a separate parameter returned by Google (encrypted, HMAC'd) rather than in the session cookie, making the callback stateless w.r.t. the cookie. Then `SameSite=Strict` works for the main session cookie because the state validation doesn't rely on the cookie being present during the callback.

### Q5: The RP receives a valid `id_token` from Google, but the `sub` in the token belongs to an active user who has a different email than what's in the RP's database. What do you do?

The `sub` is Google's permanent, stable identifier for a user. It never changes. The email can change (user changes their Gmail address). The correct behavior:

1. Look up the user by `sub` (not email). If found, this is the same person.
2. Update the email in the RP's DB to match the new email from the `id_token`.
3. Do NOT look up by email first — if you do, and the old email now belongs to a different Google account, you'd log the wrong person in.
4. Log an event: `user_email_changed` for audit trail.
5. Optionally: Send a notification to the old email address.

This is why using `sub` as the canonical foreign key to Google's identity is critical.

---

### Q6: What is a "confused deputy" problem in the context of OAuth, and how does the `audience` (`aud`) claim in the `id_token` prevent it?

A confused deputy is when a service (the deputy) with elevated privileges is tricked into acting on behalf of an attacker because it can't distinguish who it's acting for.

In OAuth: imagine a resource server that accepts any Google-issued `id_token`. An attacker who has obtained a valid `id_token` for a different RP (by logging into that RP) can present it to the first resource server. Both tokens are signed by Google and look valid.

The `aud` (audience) claim prevents this. It specifies which `client_id` (which RP) the token was issued for. The resource server **must** validate that `aud` matches its own `client_id`. If an attacker presents an `id_token` with `aud=other_app_client_id` to the resource server with `client_id=this_app_client_id`, validation fails.

### Q7: How would you detect and respond to a refresh token compromise in production?

**Detection signals:**
- Google returns `invalid_grant` when the RP tries to use a refresh token (Google has already detected abuse and revoked it).
- Unusual API access patterns: the access token (derived from the compromised refresh token) is used from a different geographic location, at unusual hours, or at higher-than-normal velocity.
- RP's audit logs show `user.logged_in` events for a user who hasn't actively used the app in weeks (attacker periodically refreshing to maintain access).
- DLP alert: bulk data access by a user account.

**Response:**
1. Immediately revoke all refresh tokens for the affected user(s): `POST https://oauth2.googleapis.com/revoke?token=<refresh_token>` for each stored token.
2. Invalidate all active sessions for the affected user in Redis.
3. Delete stored access tokens and refresh tokens from DB.
4. Force re-authentication on next request.
5. If compromise scope is broad (DB dump): Rotate `client_secret`, notify all users to re-authenticate.
6. Notify affected users via email/SMS with guidance.
7. Investigate how the tokens were compromised (DB breach? Memory dump? Log leak?).

### Q8: The RP is deployed behind a CDN. The CDN caches the `/auth/callback` endpoint response. What are the implications and how do you prevent it?

**Implications:** The CDN serves a cached OAuth callback response (with an attached session cookie for a previous user) to a new user. The new user gets logged in as the previous user. This is a catastrophic multi-tenant session bleed.

Even without a full cache hit: if the CDN caches the `Set-Cookie` header from a callback response, it could inject a session cookie into the response of any subsequent cached request.

**Prevention:**
1. `Cache-Control: no-store, no-cache, private` on all auth-related responses. The CDN must not cache anything with a `Set-Cookie` header.
2. Configure the CDN to always pass through requests to `/auth/*` to the origin.
3. Configure the CDN to never cache responses containing `Set-Cookie` headers.
4. Use origin shield or path-based CDN routing to exclude auth paths entirely.
5. Test: After login, verify that a second browser hitting the same path does NOT get a cached session.

### Q9: What are the differences between validating the `id_token` locally vs. using Google's `tokeninfo` endpoint? When would you use each?

**Local validation (recommended for production):**
- Fetch Google's JWKS, verify JWT signature locally.
- Fast (no network call in the hot path after JWKS is cached).
- Scales independently — no dependency on Google's `tokeninfo` endpoint availability.
- Requires JWKS cache management and key rotation handling.

**`tokeninfo` endpoint (`GET https://oauth2.googleapis.com/tokeninfo?id_token=<token>`):**
- Google validates the token server-side and returns claims.
- Slower (one network round-trip to Google per login).
- Always uses Google's current keys (no key rotation concern).
- Rate-limited by Google.
- Creates a single point of failure (Google's tokeninfo must be up).
- Use case: Simple scripts, one-off tools, environments where dependency management of JWKS is impractical.

**What if JWKS unavailable during local validation?**
Fail open (dangerous) or fail closed (safe). Always fail closed — return auth failure. Log the JWKS fetch failure as a high-priority alert. This is why JWKS must be cached aggressively and refreshed proactively before expiry.

### Q10: Describe what happens if the RP sends an incorrect `redirect_uri` in the token exchange that doesn't match the one used in the authorization request.

Google binds the authorization code to the exact `redirect_uri` that was present during the authorization request. In the token exchange, the `redirect_uri` is sent again not as a functional redirect (no redirect actually happens) but as a verification parameter.

If they don't match exactly (even a trailing slash difference, HTTP vs HTTPS, different port), Google returns:

```json
{"error": "redirect_uri_mismatch"}
```

**Why this matters for security**: If Google didn't re-verify the `redirect_uri` at token exchange time, an attacker who intercepted a code sent to a different URI could exchange it by presenting any URI. By binding the code to the original URI and requiring it at exchange, the code is only useful to an entity that received it at the correct URI — tying possession of the code to the legitimate redirect path.

### Q11: How does PKCE protect against code interception when the `client_secret` is already supposed to protect the token exchange?

PKCE solves a different problem from the `client_secret`. The `client_secret` authenticates the RP as an entity — it proves "this is Example App Inc." But it doesn't prove "this specific token exchange request is from the same browser session that started the flow."

PKCE is a per-flow proof. The `code_verifier` is generated fresh for each OAuth flow and never leaves the device that started the flow. Only the hash (`code_challenge`) is sent to Google.

Scenarios where PKCE adds value even with `client_secret`:
- **Log-based code theft**: An attacker reads the authorization code from RP server logs (the code appears in the URL). The attacker has the `client_id` (public) but not the `code_verifier` (stored in user's session). Without PKCE: attacker can exchange using the `client_secret` (if also leaked). With PKCE: attacker is blocked even with both the code and the `client_secret`.
- **Public clients** (mobile apps, SPAs): No `client_secret` at all. PKCE is the only proof of legitimate exchange. This is why PKCE is mandatory for public clients per RFC 9700.
- **Server-side MITM before TLS**: In corporate networks with TLS inspection proxies, PKCE provides an additional layer.

### Q12: What is token binding (RFC 8471) and why is it not widely deployed despite being a strong security control?

Token binding cryptographically ties a token (access token, refresh token) to the TLS connection on which it was issued. The client generates an asymmetric key pair, signs a binding message with the private key, and sends the public key to the server during the TLS handshake. The server binds the token to that public key. On subsequent uses, the client must prove possession of the private key, making stolen tokens useless (a stolen access token cannot be used from a different TLS connection without the matching private key).

**Why not widely deployed:**
1. **Requires support at every layer**: TLS library, browser, web server, application — all must implement it. The spec requires TLS-layer hooks that most TLS libraries don't expose.
2. **Browser support removed**: Chrome shipped token binding support, then removed it (2021) due to lack of ecosystem momentum.
3. **Complexity**: Key management, key rotation, device binding (the key can't follow the user across devices).
4. **Alternative mitigations suffice for most threat models**: PKCE + short-lived tokens + refresh token rotation achieve most of the security benefit with far lower implementation complexity.
5. **DPoP (Demonstration of Proof-of-Possession)** — RFC 9449 — is the modern successor. DPoP is application-layer (JWT-based), easier to implement, and gaining traction in OAuth 2.1.
